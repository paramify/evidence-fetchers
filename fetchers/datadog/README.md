# DataDog Evidence Fetchers

This directory contains 11 Python fetchers that collect evidence from the DataDog API for FedRAMP 20x compliance (Class C / Moderate). Each fetcher is self-contained, reads credentials from environment variables, and writes a JSON evidence file to `$OUTPUT_DIR/<fetcher_name>.json`.

## Setup

See `API_KEY_SETUP.md` for service account creation, API key generation, and required permissions.

Required environment variables:

| Variable | Description |
|---|---|
| `DATADOG_API_KEY` | DataDog API key |
| `DATADOG_APP_KEY` | DataDog application key |
| `DATADOG_BASE_URL` | API base URL (default: `https://api.ddog-gov.com` for GovCloud) |
| `DATADOG_SIGNALS_LOOKBACK_DAYS` | Lookback window for SIEM signals (default: 30) |
| `DATADOG_INCIDENTS_LOOKBACK_DAYS` | Lookback window for incidents (default: 90) |

Override `DATADOG_BASE_URL` to `https://api.datadoghq.com` for commercial DataDog.

## Running

```bash
# Run all DataDog fetchers via the orchestrator
python 3-run-fetchers/run_fetchers.py

# Run a single fetcher standalone (reads from .env)
python fetchers/datadog/datadog_siem_detection_rules.py

# Run with explicit output directory
python fetchers/datadog/datadog_siem_detection_rules.py --output-dir /tmp/evidence
```

## Output Format

Every fetcher produces a JSON file with this envelope:

```json
{
  "status": "success | partial_or_empty | error",
  "api_endpoint": "https://api.ddog-gov.com/...",
  "record_count": 42,
  "data": [...],
  "analysis": { ... },
  "retrieved_at": "2026-05-21T00:00:00Z"
}
```

The `analysis` block contains scalar fields and summaries targeted by `validationRules` in the catalog.

## Fetcher Reference

### `datadog_siem_detection_rules`

Retrieves all custom (non-default) SIEM detection rules.

**API:** `GET /api/v2/security_monitoring/rules?is_default=false`  

**Key analysis fields:**
- `total_rules`, `enabled_rules`, `disabled_rules`
- `rules_updated_last_90_days` — proves rules are actively maintained
- `rules_by_type`, `rules_by_severity`

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-MLA-OSM | Operating SIEM Capability | **Strong** — custom enabled rules with recent `updatedAt` timestamps prove active configuration |
| KSI-MLA-LET | Logging Event Types | **Strong** — `rules_by_type` and `rules_by_severity` enumerate event categories under detection |

---

### `datadog_siem_signals`

Retrieves security signals generated by detection rules within a configurable lookback window.

**API:** `POST /api/v2/security_monitoring/signals/search`  

**Key analysis fields:**
- `total_signals`, `severity_counts`, `status_counts`
- `top_triggering_rules` — top 10 rules by signal volume
- `lookback_days`, `date_range`

> **Privacy:** `message` field is not collected — signal messages contain PII (usernames, IPs) from live security events.

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-MLA-OSM | Operating SIEM Capability | **Strong** — active signals in a bounded date range prove the SIEM is processing events continuously |
| KSI-MLA-RVL | Reviewing Logs | **Strong** — signal status lifecycle (open → under_review → archived) is direct evidence of persistent log review |

---

### `datadog_siem_configuration`

Retrieves SIEM operational state: suppression rules and notification integrations.

**APIs:** `GET /api/v2/security_monitoring/configuration/suppression_rules`, `GET /api/v1/integration/webhook/configuration/webhooks`

**Key analysis fields:**
- `suppression_rules_count`, `enabled_suppression_rules`
- `notification_integrations_count`, `notification_types` (e.g. `["pagerduty", "slack"]`)

> **Privacy:** Only the domain of webhook URLs is stored, never the full URL.

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-MLA-OSM | Operating SIEM Capability | **Solid** — notification integrations prove SIEM alerts reach humans; suppression rules with expiration dates prove controlled noise reduction |

---

### `datadog_log_pipelines`

Retrieves log processing pipeline configurations.

**API:** `GET /api/v1/logs/config/pipelines`  

**Key analysis fields:**
- `total_pipelines`, `enabled_pipelines`, `disabled_pipelines`, `read_only_pipelines`
- `disabled_pipelines_with_zero_processors` — flags decommissioned pipelines
- `pipelines_summary` — name, enabled state, processor count, filter query per pipeline

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-MLA-LET | Logging Event Types | **Strong** — `pipelines_summary` filter queries name each ingested source (CloudTrail, Okta, Kubernetes audit) explicitly |
| KSI-CMT-LMC | Logging Changes | **Solid** — CloudTrail pipeline with active processors proves infrastructure changes are ingested and processed |

---

### `datadog_log_indexes`

Retrieves log index retention configurations.

**API:** `GET /api/v1/logs/config/indexes`  

**Key analysis fields:**
- `total_indexes`, `min_retention_days`, `max_retention_days`, `indexes_below_90_days`
- `min_retention_days_audit_indexes` — minimum retention across indexes scoped to `cloudtrail`, `okta`, or `kubernetes.audit`
- `audit_indexes_below_90_days` — count of audit-critical indexes below the 90-day FedRAMP threshold
- `indexes_summary` — name, retention days, filter, daily limit per index

> **Compliance note:** FedRAMP requires ≥ 90 days retention. `indexes_below_90_days` flags all indexes; `audit_indexes_below_90_days` is the compliance-relevant check (debug/dev indexes with short retention are expected and do not constitute a finding).

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-MLA-LET | Logging Event Types | **Strong** — each index filter query is a machine-readable inventory of what event types are retained and for how long |
| KSI-MLA-RVL | Reviewing Logs | **Solid** — retention config is the infrastructure prerequisite for persistent log review |

---

### `datadog_log_archives`

Retrieves long-term log archive configurations (S3, Azure, GCS destinations).

**API:** `GET /api/v2/logs/config/archives`  

**Key analysis fields:**
- `total_archives`, `enabled_archives`, `archives_by_destination_type`
- `archives_summary` — name, state, destination type, bucket, and path per archive

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-MLA-OSM | Operating SIEM Capability | **Solid** — S3 archives make the SIEM tamper-resistant; logs in S3 cannot be retroactively altered via the DataDog console |
| KSI-RPL-ABO | Aligning Backups with Objectives | **Solid** — multiple enabled archives with separate paths for general vs. security logs demonstrates backup segmentation |

### `datadog_monitors_list`

Retrieves monitor configurations (metric alerts, log alerts, service checks).

**API:** `GET /api/v1/monitor`  

**Key analysis fields:**
- `total_monitors`, `monitors_by_status`, `monitors_by_type`
- `monitors_in_alert_count` — monitors currently firing

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-MLA-LET | Logging Event Types | **Strong** — each monitor with its query is a machine-readable statement of what event types are under continuous automated observation |
| KSI-CNA-OFA | Optimizing for Availability | **Strong** — availability monitors (CPU, RDS connections, K8s node readiness) with `notify_no_data` prove persistent infrastructure health observation |

### `datadog_agent_hosts`

Retrieves real-time inventory of all hosts reporting via the DataDog agent.

**API:** `GET /api/v1/hosts`  

**Key analysis fields:**
- `total_hosts`, `active_hosts`, `muted_hosts`
- `hosts_by_platform`, `hosts_by_cloud_provider`
- `agent_version_distribution`, `hosts_with_outdated_agent` — count of hosts not on the fleet's most common agent version

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-PIY-GIV | Generating Inventories | **Strong** — DataDog agent is an authoritative real-time source; `last_reported_time` proves inventory currency |
| KSI-MLA-EVC | Evaluating Configurations | **Partial** — agent presence proves config management tooling is deployed, but not that configuration checks are passing (gap: needs `datadog_infra_agent_checks` fetcher via `GET /api/v1/check_run`) |

---

### `datadog_apm_services`

Retrieves the APM service catalog — registered services with ownership and tier metadata.

**API:** `GET /api/v2/services/definitions`  

**Key analysis fields:**
- `total_services`, `services_by_language`, `services_by_team`
- `services_with_owner`, `services_without_owner`

> **Privacy:** Raw contact details (email addresses) are not collected. Each service record stores `has_contact: bool` and `contact_types: list` only.

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-PIY-RSD | Reviewing Security in the SDLC | **Solid** — service catalog with owners, tiers, and contact types proves SDLC governance structure required by CISA Secure By Design |
| KSI-MLA-LET | Logging Event Types | **Partial** — provides the authoritative list of services (the "information resources"); full coverage requires correlating against `log_indexes` and `log_pipelines` |

---

### `datadog_incidents_list`

Retrieves incident records within a configurable lookback window for pattern analysis and after-action review.

**API:** `GET /api/v2/incidents`  

**Key analysis fields:**
- `total_incidents`, `incidents_by_severity`, `incidents_by_status`
- `incidents_with_postmortem`, `resolved_without_postmortem_ids` — IDs of resolved incidents missing a postmortem
- `avg_resolution_hours`, `lookback_days`

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-INR-AAR | Generating After Action Reports | **Solid** — `postmortem_id` is machine-readable proof that AARs exist; `resolved_without_postmortem_ids` surfaces the gap |
| KSI-INR-RPI | Reviewing Past Incidents | **Solid** — severity/status distribution across a 90-day lookback provides an auditable incident history with resolution metrics |

---

### `datadog_incident_timelines`

Retrieves per-incident timeline cells to prove after-action reports are substantive and IR procedures are followed.

**API:** `GET /api/v2/incidents/{id}/relationships/timeline_cells` (one call per incident)

**Key analysis fields:**
- `incidents_with_timeline`, `total_timeline_cells`
- `avg_cells_per_incident` — averaged over incidents that have at least one cell
- `cell_type_distribution`

> **Privacy:** Timeline cell `content` is not collected — incident notes in production contain hostnames, version numbers, and potentially sensitive details. Structural metadata (cell type, timestamps, author IDs) is sufficient to prove the IR process was followed.

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-INR-AAR | Generating After Action Reports | **Solid** — timeline structure (detection → root cause → resolution phases) with multiple author IDs proves AARs are substantive and multi-party reviewed |
| KSI-INR-RIR | Reviewing Incident Response Procedures | **Solid** — consistent timeline structure across incidents is evidence that documented IR procedures are being followed |

---

### `datadog_containers`

Retrieves a real-time inventory of all containers across the organization, including image, state, host/node, and Kubernetes metadata. Pairs with `datadog_agent_hosts` to provide a complete two-layer inventory: nodes → containers.

**API:** `GET /api/v2/containers`

**Key analysis fields:**
- `total_containers`, `containers_by_state`, `containers_by_environment`, `containers_by_namespace`
- `unique_images` — count of distinct image names across all containers
- `containers_without_image_digest` — containers that cannot be traced to a versioned image (primary compliance check)
- `non_running_containers_count`, `prod_non_running_containers` — production containers in a degraded state

> **Privacy:** Container environment variables are not collected. Any tag value containing `=` (indicative of an env var accidentally surfaced as a tag) is stripped before storage.

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-PIY-GIV | Generating Inventories | **Strong** — real-time container inventory from an authoritative source |
| KSI-MLA-EVC | Evaluating Configurations | **Solid** — `image_digest` proves containers are running known, versioned images |
| KSI-CNA-EIS | Enforcing Intended State | **Solid** — `state` across all prod containers proves runtime matches intended state |
| KSI-CMT-RMV | Redeploying vs Modifying | **Partial** — immutable image digests support the "redeployment not modification" principle |

---

### `datadog_infra_agent_checks`

Retrieves agent check results across the entire host fleet, proving that configuration evaluation is actively running and passing — not just that agents are deployed.

**API:** `GET /api/v1/check_run`  

**Key analysis fields:**
- `total_check_runs`, `checks_by_status` (ok/warning/critical/unknown)
- `unique_check_names` — number of distinct checks running across the fleet
- `hosts_with_critical_checks`, `hosts_with_warning_checks` — the primary validation targets
- `checks_by_name` — per-check breakdown by status (e.g. `{"ntp.in_sync": {"ok": 4}}`)

**Validation rules:**
- `hosts_with_critical_checks == 0` → PASS (no hosts with failing config checks)
- `total_check_runs >= 1` → PASS (checks are actually running)

**KSI coverage:**

| KSI | Name | Coverage |
|---|---|---|
| KSI-MLA-EVC | Evaluating Configurations | **Strong** — check results per host are direct evidence that configuration is being persistently evaluated and tested, closing the gap left by `agent_hosts` alone |
| KSI-CNA-EIS | Ensuring Infrastructure Stability | **Strong** — `hosts_with_critical_checks` and `checks_by_name` surface failing infrastructure checks (disk, NTP, CPU, memory) that indicate stability risk |
| KSI-SVC-ACM | Assessing Configuration Management | **Solid** — fleet-wide check coverage across multiple check types proves configuration management tooling is operational, not just installed |
